## 1. Key Decisions (with reasoning)
- **GO decision**: 0 Critical/High bugs open. All 10 delta ACs + 3 living ACs (AC-7-010/011/019)
  independently verified by code review, test review, and CLI smoke tests. 285 tests pass (QA-
  independent run), 98.55% coverage. R1 ordering and StateError propagation confirmed
  structurally via AST (not just by trusting test outcomes).
- **SHALLOW_TC-001 not blocking**: `test_delta_disabled_byte_identical_to_v1` proves
  determinism of the False-path but does not directly compare False vs True over the same empty-
  state scenario. Accepted because: (a) `test_delta_mark_seen_count_invariant_both_modes`
  already runs both modes over the same mixed items, and (b) code review confirms the
  `extend(new if config.delta_enabled else items)` line is the only delta conditional.
  Logged as a future test-improvement, not a blocker.
- **Dep audit clean**: `pip-audit` found no known vulnerabilities.

## 2. Contentious Points
- **AC-V2-001-003 [ASSUMED] status**: the Protocol-unchanged AC was tagged ASSUMED in S2
  because it is a correctness-by-absence claim (no method added). QA verified by code review of
  `ports.py` (only `load`/`save` present) and by `test_delta_state_store_protocol_unchanged`
  which asserts `is_seen`/`mark_seen` are not in `dir(StateStore)`. Confident it passes.
  WATCH: S6 release notes should document that the `StateStore` Protocol was NOT changed —
  any future third-party adapter does not need to implement `is_seen`.
- **test_delta_disabled_byte_identical_to_v1 (SHALLOW_TC-001)**: borderline — proving
  determinism is valuable but the AC literally says "byte-identical to a V1 run." The claim is
  satisfied by code structure + the count invariant test, not by a direct cross-mode comparison.
  Future improvement: add a variant that runs `delta_enabled=False` with `is_seen=False`
  (empty-state) and compares to `delta_enabled=True` with `is_seen=False` — both should render
  the same digest.

## 3. Implicit Assumptions
- Smoke tests hit `AuthError` before any state/delta logic because the test token is fake — this
  is intentional and appropriate for a config-layer smoke. The actual delta suppression behavior
  can only be end-to-end tested with a real token and real state, which is an S6 / post-deploy
  operator verification item.
- `render([])` returning the "no new items" doc and delivery-never-suppressed rely on contracts
  verified in earlier specs (AC-7-006, AC-6-020). QA verified the test exercises the real renderer
  (not mocked) and the actual string "No new items" appears in the delivered output.
- The `_is_seen` closures in the loop-based `test_delta_mark_seen_count_invariant_both_modes`
  use Python late-binding for the inner function — for this test the closure is re-defined each
  loop iteration so it correctly captures `item_seen.item_id` per iteration. QA traced the code
  and confirmed no closure bug.

## 4. Risky Areas
- **`_collect_all` R1 ordering (fragile but verified)**: the insert-before-mark_seen ordering
  is the single most load-bearing constraint in the entire change. Both R1 tripwire tests
  (`test_delta_mixed_new_and_seen_snapshot_before_mark_seen` and
  `test_delta_mark_seen_count_invariant_both_modes`) will immediately catch any reorder. S6
  should include a note in the release checklist to re-run these two tests after any future
  touch to `_collect_all`.
- **Mock-fixture convention**: all 27 `test_pipeline.py` tests now require
  `mock_state.is_seen.return_value` or `mock_state.is_seen.side_effect` to be set. A new test
  added in a future PR without this will silently pass with an empty digest. This is the one
  fragile area that could produce false-green tests. No automated lint guard exists; it is
  convention-only. Recommend adding a comment to the test file's fixture section documenting
  this requirement.
- **AC-V2-001-003 [ASSUMED]**: if a future change ever adds `is_seen` to the `StateStore`
  Protocol, the `test_delta_state_store_protocol_unchanged` assertion will fail loudly — which
  is the intended early-warning behavior.
