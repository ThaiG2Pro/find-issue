## 4. Risky Areas (passed QA but fragile)

- **Lock exit-code arm ordering** (`cli.py::run`): The `LockHeldError` handler must remain the first `except` clause. A refactor that reorders or merges except arms could silently change overlap behavior from exit-0 to exit-1. This is not guarded by any structural test — only by `test_run_lock_held_exits_0_not_1`.
- **`upsert_block` newline symmetry**: The double-`\n` round-trip contract is subtle. The `remove_block` strip logic `start_idx - 1 if start_idx > 0 else start_idx` is unconditional — it always strips the separator. Any change to `upsert_block`'s separator logic without a matching change to `remove_block` would break byte-identity. The 8 parametrized tests are the guard.
- **`assert_no_secret` guards the final text** at the CLI level. If a future generator returns the text indirectly (via a wrapper that modifies it after the guard), the backstop would not catch the modification. Current architecture: guard is called on the final string before any I/O.
- **`NO_COLOR` enforcement** (`cli.py::run`): `os.environ["NO_COLOR"] = "1"` is set at CLI entry (B-002 fix). If a future change clears this env var or reads it before the `run()` function sets it, the no-color contract for non-TTY would break. Currently harmless — all output paths respect `NO_COLOR`.
