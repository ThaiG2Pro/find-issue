## 4. Risky Areas

- **`_classify_graphql` check order**: Verified correct and pinned by `test_disabled_null_shape_detected_BEFORE_errors_raise`.
  This is the highest-risk area of the change. If this test is deleted or the implementation is
  refactored, the null-shape check MUST remain before the errors-raise check. The comment "This check
  MUST precede..." in the source is load-bearing documentation.
- **Inner-guard `isinstance` exclusion for discussions**: Tested by two dedicated tests (AuthError +
  RateLimitError). The pattern (two-arm catch with fatal re-raise) is idiomatic from v2-003 memory
  lessons. It is correct but easy to regress if someone "simplifies" the guard to a single tuple.
  Two arms MUST be kept separate.
- **R1 invariant**: `_partition_new` before `mark_seen` is tested by `test_discussion_fetch_failure_issues_survive`
  asserting exact `mark_seen` call args. This test is the only regression guard for the concatenation
  order. If `discussions` were not added to `items` before `_partition_new`, the partition would
  miss them and mark_seen would also miss them — a silent idempotency failure.
- **Token in POST body**: The GraphQL `json_body` dict is constructed in `fetch_discussions` from
  the `_DISCUSSIONS_QUERY` constant + owner/name/cursor. There is no mechanism for the token to
  enter this dict. The test using a token sentinel string confirms the raw POST bytes are clean.
