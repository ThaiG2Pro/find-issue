## 2. Contentious Points (Borderline / SPEC-UNCLEAR)

- **BUG-1 (feature vs bug borderline)**: The shell chain `git diff --cached --quiet || git commit ... && git push`
  has `git push` running even when diff is quiet, because bash parses `A || B && C` as `(A || B) && C`
  left-to-right (not right-to-right as the dev report assumed). Dev and ADR-001 intended
  `A || (B && C)`. In practice the effect is benign (push fast-forwards, exits 0), so this was
  classified LOW rather than Medium. One could argue it's a design document error (S3) not a code
  bug — the YAML matches what was intended-but-mis-analyzed.

- **if: always() on persist**: The developer self-review flagged this as [HIGH] concern. QA confirms
  the behavior is correct: a failed `osspulse run` that produces no state file → `git add -f` on
  absent path → nothing staged → `git diff --cached --quiet` exits 0 → loop short-circuits → no
  commit, no empty-commit error, step exits 0. Intentional and safe.

## 3. Implicit Assumptions (Test Limitations)

- **No live CI run**: The workflow was verified via YAML parse and static code review only.
  A real GitHub Actions run could reveal secrets-not-set errors or runner timing issues that
  static review cannot catch. The operator must perform a `workflow_dispatch` smoke test after
  first secrets configuration.
- **`config.toml` operator setup**: This CR does not commit a real `config.toml`. The example
  file exists; the operator must `git add -f config.toml` before first run. This is documented
  in README but not testable by QA in this environment.
- **`git push` success assumption**: The persist step's push assumes the `GITHUB_TOKEN` from
  the checkout step has write access. If the repo is a fork (read-only token), push will 403.
  Low risk for a self-hosted personal tool.

## 4. Risky Areas (Passed But Fragile)

- **Shell operator precedence (BUG-1)**: The `||`/`&&` chain is not wrong in practice (push
  no-ops when clean) but it is subtler than intended. If the script is extended (e.g. adding
  `etags.json` to the persist step), a developer might add more `&&` terms assuming
  right-associativity and introduce a real bug. Consider adding a `# NOTE: (A||B)&&C` comment
  or fixing the subshell `(...)` grouping for clarity.
- **`if: always()` on persist + absent state.json**: Verified safe by reasoning, but not by
  running a deliberately failed digest job. If `osspulse run` is changed to write a partial
  state.json before crashing, the partial file would be committed. This is a future regression
  risk, not a current bug.
