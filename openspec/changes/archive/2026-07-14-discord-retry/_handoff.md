## 1. Key Decisions (GO/NO-GO reasoning)

**GO.** 0 bugs found across 17 ACs, 81 tests, 94% coverage.

The three critical invariants demanded by the change were independently verified in source code:

1. **URL leak (T1)**: All three `error_msg` construction paths in `_do_post_with_retry` use only
   `HTTP {status}`, `{type(exc).__name__}`, or `timed out after {self._timeout}s` — none reference
   `self._webhook_url`, `str(exc)`, or `repr(request)`. Two dedicated post-retry secrecy tests
   (`test_post_retry_url_secrecy_http`, `test_post_retry_url_secrecy_timeout`) each assert
   both full URL and token substring are absent.

2. **Backoff formula**: `backoff = self._backoff_base * (2**attempt)` with `attempt` starting at
   `0`; `wait = max(retry_after, backoff) if retry_after is not None else backoff`. Exact formula
   per ADR-002 and AC-001-006. Sequence `[1.0, 2.0, 4.0]` pinned in test.

3. **Non-transient 4xx immediate**: `transient = status == 429 or 500 <= status <= 599` correctly
   classifies 400/401/403/404 as non-transient → raises `DeliveryError` after exactly one POST with
   no sleep.

## 3. Implicit Assumptions (test limitations)

- **Production `httpx.Client` path not tested** (lines 412-413, 422-423): All unit tests inject
  a mock client. The `with httpx.Client(timeout=self._timeout)` branches are not exercised.
  This is a structural gap in unit tests, not a code bug — the timeout is passed correctly in
  every `client.post(...)` call. Integration test with a real/stub HTTP server would cover this.
- **No `pip-audit` / `safety`** tooling installed in this environment. `uv pip check` shows no
  compatibility issues. CI should add `pip-audit` for a proper CVE scan.
- **`backoff_base != 1.0` path** not independently tested (custom `backoff_base` e.g. 0.5).
  The formula is simple arithmetic (`backoff_base * 2**attempt`); risk is very low.
- **CLI integration smoke test** against a live/stub Discord webhook was not performed (no webhook
  available in this environment). The delivery layer is fully unit-tested.

## 4. Risky Areas (passed but fragile)

- **Coverage lines 412-413 / 422-423**: The `else: with httpx.Client(...)` branches inside
  `deliver()` for both embed and plain-text production paths are not covered. If a refactor
  changes how the internal client is created (e.g., adds connection pooling), these paths will
  silently regress. A future integration test (or a test that passes `client=None`) would close
  this gap.
- **`_parse_retry_after` `math.isfinite`**: The `inf`/`nan` guard works correctly but has no
  test. A malformed Discord response with `Retry-After: inf` or `Retry-After: nan` would be
  handled silently (returns `None`, falls back to backoff). Low risk in practice.
- **Embed-path exhaustion**: `_post_one_embed` → `_do_post_with_retry` with `noun="discord embed
  delivery"` and `unit="batch"`. The retry loop is shared; no risk of drift. But the embed path
  has only one retry-scenario test (`test_embed_parity_retry_then_success`). A
  TimeoutException-exhaustion test for embed would increase confidence.
